# 构建AI驱动的二进制分析编排系统

> 本文记录一种将大语言模型（LLM）与专业二进制分析工具链结合的架构实践——**ida-pro-analysis**。该系统的核心思想是：让 AI 充当"分析编排器"（Orchestrator），通过标准化的工具脚本间接操作专业分析引擎，实现从自然语言到结构化分析结果的闭环。文章重点阐述设计哲学与架构决策，辅以关键实现细节，供安全研究人员参考和复现。

---

## 一、问题背景

现代二进制分析工具（如 IDA Pro、Ghidra 等）功能强大，但操作门槛高：分析师需要记忆大量 API、手动执行重复性操作、在反汇编/反编译视图间反复切换。与此同时，大语言模型具备理解自然语言和推理分析的能力，却无法直接操作专业工具。

这两者之间存在一个明确的鸿沟：**AI 有推理能力但无工具操作能力，工具链有操作能力但无自主推理能力。**

ida-pro-analysis 的设计目标就是填补这条鸿沟——构建一套"AI 可调用的标准化工具接口"，让 AI 模型作为编排器，根据用户自然语言描述的分析需求，自主选择并调用合适的工具操作，最终输出结构化的分析结果。

## 二、核心架构：三层分离

系统采用**严格分层**的架构设计：

```
┌─────────────────────────────────────────────────┐
│            Layer 0: AI 编排器（Prompt）            │
│  ida-pro-analysis.md / ida-pro-analysis-evolve.md │
│  职责：理解需求 → 制定计划 → 调用工具 → 推理总结      │
└──────────────────────┬──────────────────────────┘
                       │ idat -A -S"script.py"
                       ▼
┌─────────────────────────────────────────────────┐
│         Layer 1-2: 基础设施 + 业务工具              │
│  _base.py（日志、环境变量、JSON 输出、headless 模板）  │
│  _utils.py（thunk 追踪、数据读取、地址解析）          │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│             Layer 3: 操作脚本                     │
│  query.py（13 种查询类型）                         │
│  update.py（4 种更新操作）                         │
│  scripts/（沉淀脚本库，按需扩展）                    │
└─────────────────────────────────────────────────┘
```

**关键约束：依赖方向严格单向** `_base.py ← _utils.py ← query.py / update.py`。禁止反向依赖和循环依赖。这条规则确保了每一层可以独立测试和演进，而不会因上游变更引发连锁故障。

## 三、设计哲学

### 3.1 "AI 不碰 GUI，只走 CLI"

这是整个系统最核心的设计决策。IDA Pro 的 GUI 是为人类设计的交互界面，AI 无法也不应直接操作 GUI。系统的做法是利用 IDA Pro 提供的命令行工具 `idat`（IDA 的 headless 模式），通过 `-A -S"script.py"` 参数让 `idat` 加载二进制文件后自动执行指定脚本。

这意味着所有的分析操作都被封装为"无头（headless）"模式：脚本启动 → 等待分析完成 → 执行业务逻辑 → 输出 JSON 结果 → 退出进程。AI 只需要构造正确的命令行调用，就能获取结构化的分析数据。

```bash
IDA_QUERY=decompile IDA_FUNC_ADDR=main IDA_OUTPUT=/tmp/result.json \
  /opt/ida/idat -A -S"/path/to/query.py" -L/tmp/log.txt target.i64
```

### 3.2 "数据与代码分离"

运行时产生的所有中间文件、分析结果、日志都不写入项目目录，而是统一存放在 `~/bw-ida-pro-analysis/workspace/<timestamp>/` 下。项目仓库只包含版本化的脚本代码，不包含任何运行时产物。

这样做的好处是：
- 仓库保持干净，不影响版本控制
- 多次分析的产物互不干扰
- 用户的项目目录不会被分析工具污染

### 3.3 "渐进式披露"（Progressive Disclosure）

AI 编排器的 prompt 是系统中非常宝贵的资源——它决定了 AI 的"注意力"分配。如果将所有规则、策略、细节都塞进主 prompt，不仅会占用大量上下文窗口，还会分散 AI 的注意力，导致分析质量下降。

系统采用"渐进式披露"策略：主 prompt（`ida-pro-analysis.md`）控制在 **450 行以内**，只包含核心规则和触发条件。特定场景才需要的详细策略（如加壳处理、脚本生成规则）被提取到独立的"知识库"文件中，AI 仅在检测到特定信号时才通过 Read 工具按需加载。

```
主 prompt（< 450 行）：
  - 核心角色定义
  - 工具清单与调用模板
  - 执行流程与安全规则
  - "当检测到加壳 → 读取 packer-handling.md"

知识库（按需加载）：
  - packer-handling.md（200+ 行，仅在加壳场景下加载）
  - script-generation.md（150+ 行，仅在需要生成新脚本时加载）
```

### 3.4 "沉淀而非膨胀"

系统不会为了"完整性"而添加功能。每个新增的工具脚本、每条新增的 prompt 描述都需要经过"四维度量"评估：是否能减少上下文占用、减少对话轮次、提升分析速度、提升结果准确度。

满足条件的脚本经过验证后"沉淀"到 `scripts/` 目录，并注册到 `registry.json`。不满足条件的方案被明确拒绝——比如"给简单 XOR 解密写脚本"，因为 AI 自己就能完成这类计算，额外脚本只会增加系统复杂度。

## 四、关键技术实现

### 4.1 Headless 自动化：.py + .sh 二层结构

IDAPython 脚本依赖专有模块（`ida_funcs`、`ida_kernwin` 等），不能直接用 `python` 执行，必须由 `idat` 加载运行。系统将每个分析能力拆分为两层：

| 层 | 职责 | 运行环境 |
|---|---|---|
| `.py` | 业务逻辑（查询、更新） | IDA 内部（idat 加载执行） |
| `.sh`（可选） | 路径解析、环境检测、命令构造 | 终端 bash |

`.py` 脚本同时支持三种运行模式，通过 `ida_kernwin.cvar.batch` 自动检测：

```python
_batch = bool(ida_kernwin.cvar.batch)  # True = headless
_env = _parse_env_args()                # 从环境变量读参数

if _batch and _env is not None:
    _run_headless(...)                  # 等待分析 → 执行 → qexit
elif _batch:
    ida_pro.qexit(1)                    # headless 但参数缺失
elif __name__ == "__main__":
    # GUI 模式：CLI 参数解析或弹出对话框
    ...
```

**关键细节**：headless 入口逻辑必须在模块级执行，不能放在 `if __name__ == "__main__"` 内。原因是 IDA 通过 `exec(code, g)` 执行 `-S` 指定的脚本，此时 `__name__` 是脚本文件名而非 `"__main__"`。

### 4.2 公共基础设施：_base.py

所有工具脚本共享同一套基础设施，避免重复实现：

```python
# _base.py 提供的核心函数
log(msg)                              # 统一日志（headless 写 stderr，GUI 写 Output 窗口）
env_str(key, default)                 # 读取环境变量
write_json_output(output_path, result) # JSON 输出到文件
run_headless(business_func)           # headless 入口模板
```

`run_headless` 封装了所有 headless 脚本必须执行的固定流程：

```python
def run_headless(business_func):
    if not is_batch:
        return  # 非 headless 模式不执行
    ida_auto.auto_wait()      # 等待 IDA 完成自动分析
    result = business_func()   # 执行具体业务逻辑
    write_json_output(...)     # 输出 JSON 结果
    ida_pro.qexit(exit_code)   # 必须退出，否则进程不会终止
```

每个工具脚本只需要实现一个 `business_func`，返回一个包含 `success` 字段的字典，其余基础设施全部由 `_base.py` 处理。

### 4.3 查询与更新：标准化的工具接口

系统提供 13 种查询类型和 4 种更新操作，覆盖二进制分析的常见需求：

**查询类型**（`query.py`）：

| 类型 | 用途 | 特色 |
|------|------|------|
| `entry_points` | 枚举入口点 | 智能识别 EXE/DLL/SO 格式 |
| `functions` | 按模式搜索函数 | 支持通配符 |
| `decompile` | 反编译为 C 伪代码 | 自动追踪 thunk 链到真实函数 |
| `disassemble` | 反汇编指令 | 自动追踪 thunk 链 |
| `xrefs_to/from` | 交叉引用 | 支持地址和函数名 |
| `strings` | 搜索字符串及引用位置 | 子串匹配 |
| `imports/exports` | 导入导出表 | 全量列举 |
| `segments` | 段信息 | 含异常信号标注 |
| `read_data` | 读取全局数据 | 自动/string/bytes/pointer 四种模式 |
| `packer_detect` | 加壳检测 | 触发后加载详细处理策略 |

**更新操作**（`update.py`）：

| 操作 | 用途 | 安全措施 |
|------|------|---------|
| `rename` | 重命名符号 | 支持 `DRY_RUN` 预览 |
| `set_func_comment` | 函数注释 | — |
| `set_line_comment` | 行内注释 | — |
| `batch` | 批量操作 | JSON 格式，支持 `DRY_RUN` |

所有操作通过环境变量传参，输出标准化的 JSON：

```json
{
  "success": true,
  "query": "decompile",
  "data": { ... },
  "error": null
}
```

### 4.4 Thunk 追踪

二进制分析中经常遇到 thunk 函数（仅包含跳转指令的中间函数）。如果分析停在 thunk 层，得到的信息毫无意义。系统在 `decompile`、`disassemble`、`func_info` 三种查询中自动追踪 thunk 链，返回真实函数的内容，并在结果中记录完整的跳转路径。

## 五、AI 编排器的工作流程

`ida-pro-analysis` 命令定义了 AI 编排器的完整行为。当用户输入分析需求时，AI 遵循以下流程：

```
1. 解析参数     → 从自然语言中提取 IDA 数据库路径 + 需求描述
2. 预检查       → 文件存在性 + 数据库锁检测（每次调用前必须执行）
3. 判断类型     → 查询型（单次调用）/ 分析型（多轮查询 + 推理）/ 混合型
4. 制定计划     → 选择工具脚本 + 构造调用参数
5. 执行         → idat 调用 → 读取 JSON 输出 → 推理分析
6. 审计         → 返回码检查、输出非空检查、结果匹配度检查
7. 持久化       → （可选）调用 update.py 更新数据库
8. 沉淀         → （可选）保存新生成的脚本到 scripts/ 目录
9. 存档         → 写 summary.json 记录本次分析过程
10. 输出        → 格式化返回分析结果（区分"事实"与"推测"，标注置信度）
```

**安全护栏**：
- 数据库锁定时立即退出，不重试
- 同一步骤连续失败 3 次则放弃
- 单次 `idat` 调用超时 300 秒
- 累计耗时上限 15 分钟
- 所有修改操作执行前必须列出预览
- 分析结果严格区分"来自数据库的事实"和"AI 推理的推测"

## 六、进化机制：ida-pro-analysis-evolve

系统的独特之处在于它有一套自我改进机制——`ida-pro-analysis-evolve` 命令。这个命令的核心思想是：**从实际分析复盘中发现高价值改进，经讨论确认后按严格质量流程实施。**

### 六阶段进化流程

```
Phase 0: 复盘分析
  回顾最近的使用过程，统计 idat 调用次数、手写脚本次数、对话轮次、失败次数，
  识别痛点并分析根因（脚本能力不足？prompt 指引不够？架构限制？）
      ↓
Phase 1: 讨论（必须等用户确认）
  对每个候选方案做"四维度度"评估：
    - 减少上下文占用（idat 调用次数减少多少）
    - 减少对话轮次（完成同类任务少了几轮对话）
    - 提升分析速度（总耗时缩短多少）
    - 提升结果准确度（错误率/遗漏率改善）
  分为"推荐做"/"可选做"/"不建议做"三级呈现
      ↓
Phase 2: 生成需求文档
  写入 docs/需求/ 目录，包含背景目标、技术方案、实现规范、验收标准
      ↓
Phase 3: 审计需求文档（循环直到连续 2 轮无新问题）
      ↓
Phase 4: 创建任务计划 + Prompt 瘦身检查（主 prompt < 450 行）
      ↓
Phase 5: 执行任务（每步语法检查 + 同步更新文档）
      ↓
Phase 6: 审计实现（独立 agent 审计 + 端到端验证）
```

### 反模式防护

进化流程中内置了反模式警告，防止"为了改进而改进"：

| 反模式 | 为什么不该做 |
|--------|------------|
| 给简单计算写脚本（如 XOR 解密） | AI 自己就能完成；脚本膨胀占用 prompt 空间，分散注意力 |
| 为了"完整性"加功能 | 用户没遇到的痛点就不是痛点（YAGNI 原则） |
| 一次性特化脚本 | 只解决特定二进制的问题，不通用 |
| 过度封装 | 重复 3 次以上再抽象 |
| 在主 prompt 中保留可提取的详细流程 | 违反渐进式披露 |

## 七、架构决策回顾

### 7.1 为什么选择 Prompt 而非代码编排？

AI 编排器的核心逻辑完全写在 Markdown prompt 中（`ida-pro-analysis.md`，488 行），而不是写成 Python 代码。这个决策的原因是：

- **灵活性优先**：prompt 可以用自然语言描述策略，调整无需重编译、重部署
- **AI 原生**：AI 模型本身就是"执行" prompt 的引擎，不需要额外的解析/编译层
- **渐进式披露天然适配**：prompt 中可以声明"当 X 发生时读取 Y 文件"，AI 自然理解条件分支

代价是 prompt 工程需要精心设计——控制行数、避免冲突规则、保持逻辑自洽。这就是为什么进化流程中有 Phase 4.5 的"Prompt 瘦身检查"。

### 7.2 为什么选择环境变量而非命令行参数传参？

`idat -S` 机制只支持指定脚本路径，不支持传参。而 IDAPython 脚本在 IDA 内运行时也无法直接访问命令行参数（`sys.argv` 被 IDA 框架接管）。环境变量是唯一可靠的跨层传参机制。

### 7.3 为什么 JSON 而非文本输出？

AI 编排器需要结构化地读取工具脚本的输出。纯文本输出需要 AI 做额外的解析（可能出错），而 JSON 格式可以让 AI 精确提取每个字段，减少误解析。

## 八、复现指南

如果要基于本文思路构建类似的系统，以下是核心步骤：

### 8.1 基础设施层

1. **实现 `_base.py`**：统一的日志函数、环境变量读取、JSON 输出、headless 入口模板
2. **实现 `_utils.py`**：thunk 追踪、地址解析、数据读取等共享业务工具
3. **确保依赖方向单向**：基础设施不依赖业务模块

### 8.2 工具脚本层

1. **实现 `query.py`**：按需实现查询类型，每种类型返回标准 JSON
2. **实现 `update.py`**：更新操作，支持 `DRY_RUN` 预览模式
3. **每种操作独立可测试**：可通过 `idat -A -S` 命令行直接验证

### 8.3 AI 编排层

1. **编写主 prompt**：定义角色、工具清单、执行流程、安全规则
2. **控制 prompt 体积**：目标 < 450 行，详细策略移至知识库文件
3. **建立知识库**：特定场景的策略文档，通过"当检测到 X → 读取 Y"机制按需加载

### 8.4 进化机制

1. **建立复盘流程**：每次分析后统计 idat 调用次数、对话轮次等指标
2. **定义四维度量**：作为评估每个改进方案的标尺
3. **实现沉淀机制**：验证通过的脚本注册到 `registry.json`，下次优先使用

## 九、总结

ida-pro-analysis 系统的核心思想可以概括为三点：

1. **AI 作为编排器，工具脚本作为执行器**——AI 不直接操作 GUI，而是通过标准化的 CLI 接口（`idat -A -S`）间接操作专业分析引擎
2. **严格分层与渐进式披露**——基础设施、业务工具、操作脚本三层单向依赖；主 prompt 精简，详细策略按需加载
3. **经验驱动的持续进化**——从实际使用复盘中发现改进点，经四维度量评估后按严格质量流程实施

这套架构不绑定特定的 AI 模型或分析工具——只要工具引擎提供 CLI 模式和脚本扩展能力，只要 AI 模型具备工具调用能力，相同的架构模式就可以复用到其他领域（如动态分析、网络协议分析等）。其价值不在于某个具体的查询操作，而在于"让 AI 自主编排专业工具"这一工程范式的可行性验证。

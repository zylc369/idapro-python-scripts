# 需求：Writeup 写作命令

## §1 背景与目标

**来源痛点**: 用户需要写安全分析 writeup（CTF 解题报告），当前没有专用命令指导写作。writeup 通常很长（680-1752 行），GLM-5.1 在生成长文档时容易卡住（试图一次 Write 整个文档导致超时/静默失败）。

**预期收益**:
- 统一 writeup 格式和质量标准
- 通过分段编辑策略消除"长文档卡住"问题
- 减少对话轮次（无需用户反复提醒格式和风格）

**改动范围**: 新增 1 个命令文件

## §2 技术方案

### 2.1 新增文件

| 文件 | 路径 | 说明 |
|------|------|------|
| writeup 写作命令 | `.opencode/commands/write-writeup.md` | writeup 写作规范 + 长文档防卡策略 |

### 2.2 不修改的文件

不修改任何现有文件。命令文件独立存在。

### 2.3 命令定位

这是一个通用命令（不属于任何特定 agent），任何 agent 在需要写 writeup 时都可以引用。放在 `.opencode/commands/` 目录下，与其他命令（`gui-interact-pc.md`、`security-analysis-evolve.md`）同级。

## §3 实现规范

### 3.1 实施步骤拆分

```
步骤 1. 创建 write-writeup.md 命令文件
  - 文件: .opencode/commands/write-writeup.md（新建）
  - 预估行数: ~150-180 行（不含注释和空行约 100-120 行）
  - 内容:
    a. 命令 frontmatter（description）
    b. writeup 格式规范（标题/元信息/章节结构）
    c. 写作风格规则
    d. 按题目类型的章节适配指南
    e. 长文档防卡策略（分段 Edit）
  - 验证点:
    1. 人工通读，确认自包含性
    2. 确认覆盖 a-e 全部内容
    3. 确认长文档防卡策略包含具体的分段步骤和行数限制
    4. 确认写作风格规则与现有 5 篇 writeup 的实际风格一致
  - 依赖: 无
```

## §4 验收标准

### 功能验收

- [ ] 命令文件存在于 `.opencode/commands/write-writeup.md`
- [ ] 包含 writeup 格式规范（标题、元信息、目录、必备章节）
- [ ] 包含写作风格规则（教学式、一句话概括、代码示例、失败记录）
- [ ] 包含按题目类型的章节适配指南
- [ ] 包含长文档防卡策略（分段 Edit、行数限制、进度追踪）

### 回归验收

- [ ] 不修改任何现有文件
- [ ] 命令文件位于正确的目录（`.opencode/commands/`）

### 架构验收

- [ ] 命令文件自包含（不依赖 agent prompt 上下文即可理解）

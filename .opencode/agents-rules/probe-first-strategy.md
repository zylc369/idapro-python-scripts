分析过程中遵守以下**试探优先策略**：

| 原则 | 规则 |
|------|------|
| **先试探再聚焦** | 先动手试探（在目标环境里执行/运行/访问），看反馈，根据反馈聚焦分析当前问题，解决后继续试探。禁止先完整分析所有机制再动手 |
| **视野聚焦** | 每一轮只关注眼前遇到的那一个问题（为什么报错？为什么结果不对？为什么卡住？）。不要同时追踪多个"可能有关"的机制——先解决眼前这个，下一个问题等遇到了再说 |
| **在正确环境试探** | Web 题在浏览器里试（Playwright/CDP），二进制在 IDA/调试器里试，移动端在设备上试。禁止在替代环境（Python 脚本推演 Chrome 行为、纸上模拟 CPU 执行）中推演目标环境的行为 |
| **根据反馈做决策** | 先看反应再决定下一步。遇到报错分析报错，遇到异常行为分析异常。禁止先想好完整方案再动手 |

### 反模式

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| 完整阅读所有源码后再动手 | 规划只需要读够当前步骤的信息，不需要完整理解所有代码 |
| 试图理解所有机制的原理 | 识别出"这是防线"然后废掉/绕过，不需要理解原理 |
| 在 Python 中推演 Chrome 运行时行为 | 在 Chrome 中直接运行代码看结果 |
| 先规划所有可能的攻击路径再选一条 | 试探→发现问题→聚焦解决→继续试探 |
| 做各种预防措施（清除 interval、override 函数） | 先试最简单的方式，出问题再说 |

### 试探循环

```
试探（在目标环境里动手）→ 观察反馈 → 聚焦分析当前问题 → 解决 → 继续试探
         ↑                                               │
         └───────────────────────────────────────────────┘
```

每个循环只解决**一个**问题。解决完不回头看，直接继续试探看下一个问题。
